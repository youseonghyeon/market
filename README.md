### ⭐️⭐️️️️️️️️️️️️ DB 구조 특징 설명 - 주문 시스템
주문 시스템은 커머스에서 가장 중요한 기능들중 하나라고 생각한다. <br/>
"회원", "상품", "결제", "배송" 시스템과 정확히 맞물려 돌아가야하고 이중 하나라도 <br/>
오류가 발생해서는 주문에 대한 정상적인 처리가 불가능하다. <br/>
이러한 복잡한 주문 시스템을 단순화하기 위해 "Cart" 라는 엔티티를 도입했다. <br/>


<img width="700" alt="스크린샷 2022-06-12 오전 11 36 17" src="https://user-images.githubusercontent.com/78669797/173211924-25a3bfac-01fe-406b-ace5-609aad792354.png"><br/>
Cart 엔티티는 상품의 주문 시스템을 단순화하여 독립성을 지키기 위해 만들어진 <br/>
모듈이라고 할 수 있고, 장바구니/주문/정산 등 시스템 전반에 사용된다.<br/>

Cart 객체가 "Account 객체의 소유인가?" 혹은 "Order 객체의 소유인가?" <br/>
에 따라서 해당 Cart가 단순히 장바구니에 올라간 상품인지 <br/>
결제가 완료됬으니 배송을 준비해야 할 상품인지를 판단하게 된다.<br/>
여기서 "소유한다." 라는 개념은 "연관관계(FK)가 맺어져있다." 의 개념과 같다.


### Cart 엔티티를 활용한 주문 순서는 다음과 같다.
- 고객이 상품을 장바구니에 담으면 Cart 객체가 생성되고 이것은 "고객(Account)" 엔티티의 소유가 된다.
- 고객이 소유한 Cart들 중 원하는 Cart들을 선택하여 주문한다.
- 선택된 Cart 객체들은 "주문(Order)" 엔티티에 소유가 되고 "회원(Account)" 엔티티의 소유는 박탈된다.
- 이러한 방식의 장점은 다음과 같다.
  - 주문을 할 때 어떠한 상품의 이름/수량/옵션 등의 명시 없이 Cart 객체의 id(PK)값으로 주문이 가능하다.
  - 주문시 HTTP Request 에서 상품의 이름/수량/옵션 등이 노출되지 않는다.
  - 주문 시스템에 문제가 발생하면 Cart 를 개별적으로 추적할 수 있다.
